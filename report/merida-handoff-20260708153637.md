The new design shape is:

- `Capture Evidence` is a feature-owned module with one primary interface: `createCaptureEvidence(input)`.
- The module accepts both the new frame payload and the previous merged payload shape.
- The module owns frame merge, URL selection, field normalization, text limits, structured metadata extraction, content-source selection, debug summaries, and validation warnings.
- The Chrome extension is now a Source Page adapter that sends per-frame evidence plus the active tab URL instead of owning backend evidence assembly.
- The Job Posting parser now consumes normalized Capture Evidence and stays focused on deriving Job Posting fields and Job Content.

The reported `/resumes/create` insufficient-evidence failure was then investigated with the diagnosing-bugs workflow.

Changed files:

- `src/features/resumes/lib/deepseekResume.js`
- `src/features/resumes/test/deepseekResume.test.js`

The likely root cause was not the Python matching fix from the prior handoff. The new failure had `2/30 total supported requirements` but `0/13 required or responsibility requirements`, which pointed to requirement extraction/normalization: supported tools or skills could be marked as `signal`, so they did not count toward the required/responsibility generation gate.

The fix:

- Tightened the requirement extraction prompt so required vs preferred context is explicit.
- Added deterministic validation that promotes a requirement to `required` when its evidence phrase appears under required/responsibility language in Job Content.
- Preserved optional context: evidence under `Preferred Qualifications`, `Nice to have`, `Bonus`, or similar optional language remains `preferred`.
